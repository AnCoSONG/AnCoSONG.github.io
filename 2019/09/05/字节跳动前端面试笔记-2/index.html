10. TCP为什么要四次挥手

这个题目我就趁机把TCP 3次握手4次挥手都总结了吧，主要也是还没有学计网，但是马上就要面试还是需要看看。

首先放张简书上的图

这一张图就是TCP通信的全貌了。

TCP标志位

SYN：Synchronous 建立连接位，置为1则为建立连接

ACK: Acknowledgement 确认位，置为1则为确认

PSH: Push 传送位

FIN: Finish 结束位，置为1为结束

RST: Reset 重置位

URG: urgent 紧急位

Sequence Number: seq 顺序号码，第一次建立时是随机数，但之后每次发包+1（比如发一个SYN包+1，发一个FIN包+1）

Acknowledgement number: ack 确认号码，等于对方发包中的seq+1，只有ACK中需要写这个ack

TCP 3 次握手

第一次握手：客户端发送SYN数据包给服务端

客户端首先发送请求报文，将SYN位置为1，其中报文中还要包含seq序列号，这个序列号是随机生成的，发送之后，客户端状态变为SYN_SENT状态

SYN包例图

SYN ACK FIN PSH RST URG seq ack 其它
1 0 0 0 0 0 随机数，比如100 0 0
第二次握手：服务端接受SYN数据包，返回SYN+ACK包确认连接请求

服务端收到了SYN包，会发送一个确认报文，即SYN+ACK包，也就是SYN位置为1，ACK位置为1，同时，这个包的ack=seq+1，也就是SYN包中的seq值加1，这个包的seq为随机生成的序号。发送之后，服务端进入SYN_RECV状态。

SYN+ACK包样例图

SYN ACK FIN PSH RST URG seq ack 其它
1 1 0 0 0 0 随机数，比如70 SYN的seq+1,即101 0
第三次握手：客户端收到SYN+ACK包，发送ACK包

客户端接受到服务端的确认包之后，发送ACK确认包，其中seq为第一次SYN包的seq+1.因为seq是顺序编号。ack的值是SYN+ACK包中seq+1。发送后，客户端进入ESTABLISHED状态，而服务端接受到ACK之后也进入ESTABLISHED状态。

ACK包样例图

SYN ACK FIN PSH RST URG seq ack 其它
0 1 0 0 0 0 第一个SYN包的seq+1，即101 SYN+ACK包的seq+1,即71 0

TCP三次握手可以抽象成

客户端：我要连接，你那边可以连接么？

服务端：没问题，我可以连接。

客户端：好的，那我们建立连接。

如果不确认那么服务端就不知道他们得SYN+ACK包客户端有没有收到。

TCP 4次挥手

TCP4次挥手实现断开连接，服务端和客户端都可以发起断连。这里假设由客户端发起。

第一次挥手：主动关闭方发出FIN+ACK包，进入FIN_WAIT_1

主动关闭方发出FIN包，用来关闭主动方到被动关闭方的数据传送,但是之后如果没有收到服务器的ACK包，主动方依然会发送，并且在收到被动关闭方的FIN包后也会给出回应一个ASK包。

FIN+ASK包

SYN ACK FIN PSH RST URG seq ack 其它
0 1 1 0 0 0 根据发送方上一次的seq值加一，如15 0 0
第二次挥手：被动关闭方收到FIN+ACK，发出ASK。进入CLOSE_WAIT。

被动关闭方发送ASK包给主动方，表示确认关闭。主动关闭方ASK后进入FIN_WAIT_2。

ASK包

SYN ACK FIN PSH RST URG seq ack 其它
0 1 0 0 0 0 随机数，如30 FIN+ACK包的seq+1，16 0
第三次挥手：被动关闭方发出FIN+ASK给主动关闭方。进入LAST_ACK。

发出ASK后，立刻发送一个FIN包给主动关闭方，来表示这边也不再主动发送消息。

FIN+ASK包

SYN ACK FIN PSH RST URG seq ack 其它
0 1 1 0 0 0 30 31 0
第四次挥手：主动关闭方发出ASK确认关闭

发送ACK包给被动关闭方以通知确认。之后，主动关闭方进入等待TIME_WAIT状态，等待时长为2MSL,超过之后即进入关闭。而被动关闭方收到这个ACK之后也进入关闭。

ACK包

SYN ACK FIN PSH RST URG seq ack 其它
0 1 0 0 0 0 正常的seq加1，16 FIN+ASK包seq+1,31 0

TCP 4 次挥手可以看作

A： “我没数据发送了，断开吧,你看行不？”

B：”OK,我收到了，我检查一下有没有数据“

B：”没有数据了，我也可以断开“

A：“好的，那我们断开”

11. cookie有哪些属性、大小、浏览器如何禁止别人访问cookie

cookie介绍

cookie的属性有:

  • key: 键名

  • value: 监制

  • domain: 所属域名

  • path : 所属路径,外层可以访问子路由下的cookie，而子路由无法拿到父路由

  • size: cookie的大小

  • expires: Date|Number, 这个设置时效时间，注意使JS Date对象或者数值，数值表示天数

  • max-age: long int, 这个设置最大时长，ms单位

    不设置的默认为session cookie，也就是关闭页面就消失得cookie，设置的就是持久cookie

  • secure： 这个设置安全性，secure的cookie只能通过https传输

  • HttpOnly：这个防止XSS攻击，也就是防止别人通过document对象拿到cookie，当然自己也无法通过document拿到这个cookie

    XSS攻击就是通过document拿到cookie获取别人的身份信息，然后写入自己浏览器的对应页面这样就实现了不需要密码的登录。

  • SameSite: [strict, lax, none] ，这个是用来防止CSRF，防止cookie发送给跨站点的地址。

    • 其中strict模式是严格模式，表示任何跨站点请求都不会携带cookie
    • lax是宽松模式，但是除非使用HTTP方法改变本页面或者打开别的网站，其它情况依然不会带cookie

    详见下表，来源

    | 请求类型 | 例子 | 非 SameSite | SameSite = Lax | SameSite = Strict |
    | ——— | ——————————— | ———– | ————– | —————– |
    | link | <a href="…"> | Y | Y | N |
    | prerender | <link rel="prerender" href="…"> | Y | Y | N |
    | form get | <form method="get" action="…"> | Y | Y | N |
    | form post | <form method="post" action="…"> | Y | N | N |
    | iframe | <iframe src="…"> | Y | N | N |
    | ajax | $.get('…') | Y | N | N |
    | image | <img src="…"> | Y | N | N |

这样问题就解决了，防止别人访问cookie就使用HttpOnly标记就好。

12. localStorage，sessionStorage的区别

首先介绍localStorage和sessionStorage

sessionStorage

一种临时性存储，它的生命周期是浏览器关闭，一旦浏览器关闭sessionStorage就清空，所以可以应对刷新的情况。

它的大小有5M，比Cookie的4K大，并且数据不交换到服务器，防止了过程中被获取。

localStorage

一种永久性存储，跟sessionStorage唯一的区别是这个的生命周期是永久，除非删除。

14. BFC，如何形成BFC？

BFC定义

根据MDN，BFC是Block Formatting Context，即块格式化上下文。

一个块格式化上下文，是指Web页面的可视化CSS渲染出的一部分。它是块级盒出现的区域，也是浮动层元素进行交互的区域。

说人话的话，BFC是在常规流中竖着排列的块，IFC是横着排列的。

从BFC的定义很难理解BFC到底是什么，从BFC的特性更好理解。

BFC的特性

  • BFC内部元素不会乱跑
  • 和浮动元素产生边界

这其实只是BFC规则所反映的一点规则特性，下面介绍BFC的布局规则。

BFC布局规则！！重要！！

  1. 内部的盒子会在垂直方向，一个个地放置；
  2. 盒子垂直方向的距离由margin决定，属于同一个BFC的两个相邻Box的上下margin会发生重叠；
  3. 每个元素的左边，与包含的盒子的左边相接触，即使存在浮动也是如此；
  4. BFC的区域不会与float重叠；
  5. BFC就是页面上的一个隔离的独立容器，容器里面的子元素不会影响到外面的元素，反之也如此；
  6. 计算BFC的高度时，浮动元素也参与计算。

这些规则既印证了一些问题，比如浮动，也给出了解决方案，比如浮动清楚。

如何构建BFC

  1. 根元素；
  2. float的属性不为none；
  3. position为absolute或fixed；
  4. display为inline-block，table-cell，table-caption， flow-root；# 设置为flex为FFC
  5. overflow不为visible

利用BFC规则解决问题

  1. 浮动

    浮动会使元素脱离正常文档流，导致内容超出父级（或者父级失去高度），这是因为脱离浮动之后，父级的高度计算将不会包含float元素，利用BFC规则6，将父级转变为BFC，这样就可以把float元素包含在内。

    有的时候float之后面的元素依然会无视float元素正常排版，从而造成覆盖，这时可以构造BFC利用规则4解决问题。

  2. margin重叠

    根据规则3，同一个BFC的两个相邻box上下margin会重叠，所以如果不想重叠margin可以通过给另一个盒子加一个BFC父级，这样的话就不再是同一个BFC了。

15. BFC和IFC区别

IFC定义

IFC,inline Formatting Context, 是行格式化上下文。span，display:inline都可以生成IFC

IFC的特性

  • 内部的盒子会在水平方向，一个一个放置
  • IFC的盒子，由里面最高的盒子高度来决定。
  • 当一行不够放置的时候会自动转行。

主要影响IFC内布局的css：

  • font-size
  • line-height
  • height
  • vertical-aligin

除去BFC和IFC，还有GFC和FFC。

GFC，Grid Formatting Context，网格布局格式化上下文，通过display: grid设置

FFC，Flex Formatting Context, 自适应格式化上下文，通过display: flex或inline-flex设置

17. 回流与重绘，如何避免回流

这篇文章有很好的解释：https://www.cnblogs.com/stitchgogo/p/7920828.html

HTML在渲染时，首先构建DOM tree，之后构建样式树，之后合成渲染树，开始渲染。

回流：改动了高度，宽度，布局等，使页面必须重新构建，叫做回流 reflow

重绘：改动了元素样式，只是重新绘制新的样式，叫做重绘repaint

回流必定重绘，重绘不一定回流

回流要重画页面比重回开销大

所以优化的重点就是用重绘替代回流，能重绘不回流

在浏览器中，阻止回流的方法是创建图层

就拿chrome 来说，创建一个图层要有以下其中一个条件：

1：一个dom 元素拥有 3d 或 透视变换的css 属性（persepective, transform）

2：video 标签

3：拥有3d 上下文或加速 2d 上下文的 canvas 节点

4：混合插件 flash

5：自己做的opacity 动画 或 使用一个动画 webkit 变换的元素

6：拥有 translate3d 或 translateZ 这两会使 GPU 加速的属性

18. defer和async

这个没查是真的不知道啊

这篇文章讲的非常好！图文并茂，特别棒！

总结一下：

普通script

DOM渲染->Script下载->script执行->DOMContentLoaded

script是遇到即加载，暂停dom渲染。

script执行是按照书写顺序,即使先加载完也必须要等待直到写在前面的执行完。

defer

DOM渲染、script加载->script执行->DOMContentLoaded

这里一点不同就是，script遇到后就可以开始下载，虽然下载并行了，但即使先下载好，也要等排在前面的先执行

async

DOM渲染、script加载、script执行、DOMContentLoaded

这里的另一个区别就是，script是加载完立刻执行，不分先后，也不管是否DOMContentLoaded，遇到就下载，下载完就执行，但如果太快导致DOM还没加载完就会带来一些问题，比如DOM操作无法进行等等
