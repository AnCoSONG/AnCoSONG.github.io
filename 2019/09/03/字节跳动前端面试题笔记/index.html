2. 0.1+0.2===0.3吗，为什么？

这道题我倒是知道考察的是内存中对浮点数的近似存储。

根据以前基础课学的内容，计算机内部表示数是用二进制。怎么做呢？

比如一个浮点数是采用32位存储的（JS中Number是64位的，见下题，这里只是举例说明问题），其中

  • 1位是符号位
  • 8位是指数位
  • 25位是尾数，这个直接决定了到底有多精确

首先比如，8.25，这个对应的二进制数怎么算呢？要分成整数部分和小数部分。

8对应$ 2^3 $也就是1000,而0.25则采用乘法, 0.25*2 = 0.5, 0.5*2 = 1,说明0.25*2^2得1，那么小数部分就是.01，也就是$ 2^-2 $，因此8.25就对应1000.01，这样肯定不好看嘛，所以就转换成1.00001*$ 2^3 $，这样8.25就可以存储为

上面的说法有误，并不是因为好看的关系，这个其实是ISO 754标准，采用移码去存储float

0 00000011 10000100000000000…补够25个

所以就会出现问题！

当0.1求其二进制表达时，乘2是无法乘尽的

0.1*2 = 0.2 # 0

0.2*2 = 0.4 # 0

0.4*2 = 0.8 # 0

0.8*2 = 1.6 # 1

0.6*2 = 1.2 # 1

0.2*2 = 0.4 # 0

这回是一个无限循环,因此，计算机只能近似的存储这种值

导致0.1 + 0.2 !== 0.3

如何让他们相等呢？用到Number.EPSILON，这个值在JS中是$2^-52$次，当误差小于这个值，则可以判断相等

3.Number()的存储空间是多大，如果后台发送了一个超过最大字节的数字怎们办？

JS的Number安全值是MAX_SAFE_INTEGER, $2^{53}-1$。因此大于这个数就会出现不精确的情况，对于大数问题，可以采用

  • 数组
  • 字符串

去解决。

5.跨域、jsonp原理、CORS原理

这一道题也太大了吧…不过都是去解决跨域嘛，之前做项目时也遇到了这个问题，我现在知道的方法有

  • cors
  • jsonp
  • proxy代理

只有cors是非常认真的用了，proxy代理知道了原理，就是替换请求头嘛，jsonp还不是太了解，下面详细介绍一下。

首先第一个问题当然是，什么是跨域？

跨域源自互联网的同源策略，参考该地址的说法，浏览器从一个域名的网页去请求另一个域名的资源时，域名、端口、协议任一不同，都是跨域。

那么如何解决呢？

  • CORS，Cross Origin Resource Share
  • JSONP(JSON with Padding)
  • NGINX反向代理
  • ProxyTable
  • 等等

CORS

参考自https://segmentfault.com/a/1190000017867312?utm_source=tag-newest

CORS是一个跨域资源共享方案，为了解决跨域问题，通过增加一系列请求头和响应头，规范安全地进行跨站数据传输。

请求头包括
请求头 解释
Origin Origin头在跨域请求或预先请求中，标明发起跨域请求的源域名。
Access-Control-Request-Method Access-Control-Request-Method头用于表明跨域请求使用的实际HTTP方法
Access-Control-Request-Headers Access-Control-Request-Headers用于在预先请求时，告知服务器要发起的跨域请求中会携带的请求头信息
with-credentials 跨域请求携带cookie
响应头主要包括
响应头 解释
Access-Control-Allow-Origin Access-Control-Allow-Origin头中携带了服务器端验证后的允许的跨域请求域名，可以是一个具体的域名或是一个*（表示任意域名）。
Access-Control-Expose-Headers Access-Control-Expose-Headers头用于允许返回给跨域请求的响应头列表，在列表中的响应头的内容，才可以被浏览器访问。
Access-Control-Max-Age Access-Control-Max-Age用于告知浏览器可以将预先检查请求返回结果缓存的时间，在缓存有效期内，浏览器会使用缓存的预先检查结果判断是否发送跨域请求。
Access-Control-Allow-Methods Access-Control-Allow-Methods用于告知浏览器可以在实际发送跨域请求时，可以支持的请求方法，可以是一个具体的方法列表或是一个*（表示任意方法）。

所以如何实现跨域呢？服务端在Access-Control-Allow-Origin中设置可以请求的请求头即可。携带Cookies需要前端打开with-credentials，服务端打开credentials，这个涉及到session啊等等需要cookie的地方，尽量打开。

8. JavaScript EventLoop

终于还是来到了最恶心的Event Loop. JS

只需要记一句话，“先Macro 后Micro，Macro 1，Micro 全”

怎么说？

首先JS把执行任务分为Macrotask和Microtask。

宏任务(Macrotask)

一些异步任务的回调会依次进入macrotask queue，等待后续被调用，这些异步任务包括：

  • setTimeout
  • setInterval
  • setImmediate (Node独有)
  • requestAnimationFrame (浏览器独有)
  • I/O
  • UI rendering (浏览器独有)
微任务(Microtask)

一些异步任务的回调会依次进入microtask queue，等待后续被调用，这些异步任务包括：

  • process.nextTick (Node独有)
  • Promise
  • Object.observe
  • MutationObserver

Event Loop是如何执行的呢？

  • 执行一个Marcotask，同步代码直接执行，遇到其它的异步macrotask就将callback放入macrotask queue，将异步microtask放入microtask queue

    注意！！Macrotask一定是一次只能执行一个哦！！

  • 一个macrotask完毕后，执行微任务，同步直接执行，微任务过程中也会继续判断有无其它异步微任务或异步宏任务，有就加入对应队列，但不同的是，微任务一旦开始执行，就一定会把微任务队列清空才会进行下一步

  • 接着继续执行宏任务

  • 如此循环…

这就是Event Loop。是不是并不难懂！！
